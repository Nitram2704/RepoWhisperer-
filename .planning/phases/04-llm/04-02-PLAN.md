Create `src/docgen/llm/prompt.py` with:

1. **SYSTEM_PROMPT** — a constant string used as the system message for all doc generation calls:
```
You are a technical documentation generator. You receive code snippets from a software project and produce clear, accurate Markdown documentation.

Rules:
- Write documentation based ONLY on the code provided. Do not invent features or APIs not present in the code.
- Use proper Markdown formatting: headings, code blocks with language tags, bullet lists.
- For README generation: include a project summary, installation section (if inferable), and usage examples derived from the code.
- For API documentation: document each function/class with its signature, parameters, return type, and a brief description.
- If the code is insufficient to document something fully, state what is unclear rather than guessing.
- Keep language concise and developer-focused. No marketing fluff.
```

2. **format_user_prompt(query: str, code_chunks: list[dict]) -> str** — formats the user message:
- Takes `query` (e.g., "Generate a README for this project") and `code_chunks` (list of dicts with keys: `content`, `file_path`, `name`, `chunk_type`)
- Builds a string like:
```
## Task
{query}

## Code Context
### {file_path} — {name} ({chunk_type})
```{language}
{content}
```

### {file_path} — {name} ({chunk_type})
...
```
- Truncates total output to stay under ~8000 tokens (~32000 characters as rough 4:1 estimate). If chunks exceed this budget, include as many as fit and append a note: `\n\n(Additional code chunks truncated to stay within token budget)`
- Returns the formatted string

3. **estimate_tokens(text: str) -> int** — simple heuristic: `len(text) // 4`. Used internally by format_user_prompt to enforce the ~8K token cap. Export it for testing but it is not part of the main interface.

Keep this module pure (no side effects, no SDK imports, no env var reads). It just formats strings.
  </action>

Create `src/docgen/llm/context.py` with:

1. **generate_docs(query: str, repo: VectorRepository, provider: BaseLLMProvider | None = None, n_chunks: int = 10) -> str:**
- If `provider` is None, call `get_provider()` to get the default
- Query the VectorRepository: `results = repo.query(text=query, n_results=n_chunks)`
- Extract chunk dicts from the ChromaDB result format. ChromaDB query returns `{"ids": [[...]], "documents": [[...]], "metadatas": [[...]], "distances": [[...]]}`. Map to list of dicts:
  ```python
  code_chunks = []
  if results and results.get("documents") and results["documents"][0]:
      for doc, meta in zip(results["documents"][0], results["metadatas"][0]):
          code_chunks.append({
              "content": doc,
              "file_path": meta.get("file_path", "unknown"),
              "name": meta.get("name", "unknown"),
              "chunk_type": meta.get("chunk_type", "unknown"),
              "language": meta.get("language", "unknown"),
          })
  ```
- Format prompt: `user_msg = format_user_prompt(query, code_chunks)`
- Call provider: `result = provider.generate(SYSTEM_PROMPT, user_msg)`
- Return the result string (raw Markdown)

2. **Imports:**
```python
from docgen.llm.base import BaseLLMProvider
from docgen.llm.prompt import SYSTEM_PROMPT, format_user_prompt
from docgen.llm import get_provider
from docgen.store import VectorRepository
```

Note: Import `get_provider` from `docgen.llm` (the package), NOT from `docgen.llm.context` itself — this avoids circular imports since context.py is inside the llm package.

Actually, to avoid circular import (context.py is in the llm package, and __init__.py defines get_provider), do the get_provider import inside the function body:
```python
if provider is None:
    from docgen.llm import get_provider
    provider = get_provider()
```

3. **Type hints:** Use `from __future__ import annotations` at top of file for clean type hints.

This module is the bridge between retrieval and generation. Phase 5 will call `generate_docs("Generate a README for this project", repo)` to get back Markdown text.
